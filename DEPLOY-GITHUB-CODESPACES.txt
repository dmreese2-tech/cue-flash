# Running the demo entirely on GitHub (Codespaces + Pages)

This keeps everything inside GitHub: the app runs in a Codespace attached
to this repo, and a status page hosted on GitHub Pages tells visitors
whether that Codespace is currently awake — since Codespaces don't
auto-wake on an incoming request the way a host like Render does.

## 1. Push the repo to GitHub

(Skip if already done.)

```
cd cue-flash-server
git init
git add .
git commit -m "Cue Flash"
```

Create a repo on GitHub, then:

```
git remote add origin https://github.com/<you>/cue-flash-server.git
git branch -M main
git push -u origin main
```

## 2. Start the Codespace

1. On the repo page: **Code** → **Codespaces** tab → **Create codespace on main**.
2. The included `.devcontainer/devcontainer.json` handles the rest
   automatically: it runs `npm install`, starts `node server.js`, and
   forwards port 3000 with **public** visibility.
3. Once it's up, open the **Ports** tab in the Codespace (bottom panel) and
   copy the forwarded URL for port 3000 — something like
   `https://<random-name>-3000.app.github.dev`.

If the port doesn't show as Public automatically (Codespaces port-visibility
defaults are occasionally inconsistent), right-click it in the Ports tab →
**Port Visibility** → **Public**.

## 3. Point the status page at it

Open `docs/index.html` and replace this line near the bottom:

```js
var DEMO_BASE = "REPLACE_WITH_CODESPACE_URL";
```

with your actual forwarded URL, e.g.:

```js
var DEMO_BASE = "https://your-codespace-name-3000.app.github.dev";
```

Commit and push that change.

## 4. Turn on GitHub Pages

**Settings → Pages** → Source: **Deploy from a branch** → Branch: `main`,
folder: `/docs` → Save. GitHub gives you a URL like
`https://<you>.github.io/cue-flash-server/` — that's the page to share as
"the demo." It's static and always up, even while the Codespace is asleep.

## 5. Optional: keep it awake automatically during chosen hours

Included: `.github/workflows/keep-demo-awake.yml`. It runs on a schedule
and restarts the Codespace if it's stopped, so it's usually reachable
during whatever window you set — but it can't wake up *the instant* a
visitor arrives, since that would require putting a GitHub token
somewhere a random visitor's browser could use it, which isn't safe to
do. This is the honest middle ground: mostly-up during hours you choose,
without ever running 24/7 (which would burn through your free Codespaces
quota fast).

Setup:

1. **Create a token**: GitHub → Settings → Developer settings → Personal
   access tokens → Fine-grained token → give it access to this repo with
   **Codespaces: Read and write** permission (or a classic PAT with the
   `codespace` scope).
2. **Find your Codespace's name**: `gh codespace list` (or
   `gh api /user/codespaces --jq '.codespaces[].name'`) — it looks like
   `funny-couscous-abc123`, not the display name shown in the UI.
3. In the repo: **Settings → Secrets and variables → Actions**, add two
   repository secrets:
   - `CODESPACES_PAT` — the token from step 1
   - `CODESPACE_NAME` — the name from step 2
4. Adjust the `cron` schedule in the workflow file to match the hours you
   actually want it reachable (it defaults to every 3 hours, all day).
5. You can trigger it manually any time from the repo's **Actions** tab →
   **Keep Cue Flash demo awake** → **Run workflow**, instead of waiting
   for the schedule.

## How the sleep/wake situation actually works

Unlike Render, **a Codespace does not wake itself up when someone hits its
URL.** After a period of inactivity it just stops, and stays stopped until
you manually resume it — from `github.com/codespaces`, or `gh codespace
up` from the CLI. The status page handles this honestly rather than
pretending it's a quick cold-start:

- If the Codespace is reachable, it shows **DEMO IS LIVE** with a working link.
- If it's not, it shows **DEMO IS ASLEEP** and points people at the local
  install instructions instead of leaving them stuck on a dead link.

Practically: before you want to show someone the live demo, open
`github.com/codespaces` and resume it yourself first. It'll then stay up
until the next idle timeout (configurable per-codespace, default around 30
minutes of inactivity).

**Worth doing regardless of the workflow above:** bump your personal idle
timeout up to the maximum — GitHub → Settings → Codespaces → Default idle
timeout → **240 minutes (4 hours)**. That alone means it takes far fewer
wake-ups to stay up through a demo session, and pairs well with the
scheduled workflow (which only has to nudge it back on every few hours
instead of constantly).

## Costs / limits worth knowing

GitHub Codespaces has a monthly free quota of compute and storage on
personal accounts; running this small Node app is very light, but the
Codespace still counts against that quota while it's awake — check your
current usage/limits at **github.com/settings/billing** since these
numbers do change over time.

## Custom domain for the status page (optional)

Since Pages is just a static page, you can map `apps.upstage.systems` (or
a subdomain) to it the same way as any other GitHub Pages custom domain:
**Settings → Pages → Custom domain**, then add the CNAME record it gives
you at your DNS provider.
